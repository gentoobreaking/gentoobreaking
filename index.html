Hexo Migrator for Blogger 遷移指南

從 Blogger 搬家到 Hexo 的完整實戰紀錄

1. 安裝與初始化 Hexo

參考 Hexo 官方文件 進行基本環境架設。

cd ~/git/gentoobreaking/
npx hexo init blog
cd blog
npm install
npx hexo clean && npx hexo server

2. 開啟 HTML 與搜尋支援

修改 _config.yml 以支援 Markdown 內的 HTML 標籤及本地搜尋。

# 允許 Markdown 內使用 HTML 標籤
marked:
  gfm: true
  breaks: true
  unsafe: true

# 關閉預設高亮以相容主題
highlight:
  enable: false

# 搜尋外掛配置
search:
  path: search.xml
  field: post
  content: true
  format: html

3. Blogger 文章遷移 (Node.js 腳本)

使用 JSDOM 解析 Blogger 導出的 feed.atom 並轉換為 Hexo 格式。

遷移核心邏輯

# 執行遷移腳本 (需先安裝 jsdom: npm install jsdom)
node -e "
const fs = require('fs');
const { JSDOM } = require('jsdom');

const feed = fs.readFileSync('/Users/david/Downloads/Takeout/Blogger/Blogs/IT@DOG/feed.atom', 'utf8');
const dom = new JSDOM(feed, { contentType: 'application/xml' });
const doc = dom.window.document;
const entries = doc.querySelectorAll('entry');

entries.forEach((entry, i) => {
  const title = entry.querySelector('title')?.textContent || '';
  const published = entry.querySelector('published')?.textContent || '';
  const content = entry.querySelector('content')?.textContent || '';
  const pubDate = published ? published.slice(0,10) : '1970-01-01';
  
  // 生成 Slug 並處理中文字元與特殊符號
  const slug = title.toLowerCase()
    .replace(/[^\w\u4e00-\u9fa5]+/g, '-')
    .replace(/(^-|-$)/g, '');
  
  const filename = pubDate + '-' + slug + '.md';
  const md = '---\\ntitle: ' + title + '\\ndate: ' + pubDate + '\\n---\\n\\n' + content;
  
  fs.writeFileSync('source/_posts/' + filename, md);
});

console.log('Total: ' + entries.length + ' posts');
"

4. 主題配置 (NexT)

推薦使用穩定且功能強大的 NexT 主題。

步驟 指令 / 設定
安裝主題 npm install hexo-theme-next
功能組件 npm install hexo-generator-feed hexo-generator-search --save
語系設定 將 language 設為 zh-TW 避免語系回退錯誤。

5. 常用指令總結

維護建議流程：

  1. npx hexo clean — 清除快取
  2. npx hexo g — 生成靜態檔案
  3. npx hexo s — 啟動本地預覽

6. 還原預設主題

如果需要換回 Hexo 預設的 landscape 主題，可執行以下指令：

npx hexo config theme landscape
npm uninstall hexo-theme-next

OpenClaw 深度使用心得：強大、危險，以及龍蝦化 AI 的現狀與反思

發表於 2026-06-14 更新於 2026-06-18 分類於 AI 觀點

OpenClaw 深度使用心得：強大、危險，以及龍蝦化 AI 的現狀與反思

AI Agent OpenClaw 開發工具 LLM Scrum vibe coding

作者：david（github.com/gentoobreaking）

最近花了不少時間在折騰 OpenClaw 家族的生態系。從本地 LLM 的搭配測試，到搭建 Scrum 團隊執行專案，再到踩過大大小小的坑——這篇記錄了一些實際體驗與觀察，希望對正在評估或已經在使用 OpenClaw 的人有參考價值。

一、環境搭建與工具鏈

本地模型 + 雲端 LLM 的混合配置

初期以 Ollama 和 Omlx 來連接 OpenClaw，測試了好幾個本地模型。同時也接上 OpenRouter 上的免費 AI 模型，並配置了 fallback 機制，確保 LLM 後端不會因為單一供應商故障而中斷。除此之外，也測試了兩個不同開發團隊版本的 QClaw。

關於硬體衝動：一度很想買 Nvidia DGX Spark，但冷靜想想——以 M5 128GB 的 Mac 來說，本地跑 MiniMax 2.7 和 DeepSeek V4 Flash 已經夠用了。目前還在猶豫當中。

管理介面的選擇

測試了好幾個管理介面後，目前主要使用 OpenClam-Admin。選擇它的原因有三：

  • 外觀較為美觀，視覺上更現代
  • 功能相對完善，該有的都有
  • 對話框即時性高——最重要的是，對話框內容包含了成員的思考步驟以及完整的執行指令稿，讓我更清楚每一個成員及目前所有的步驟狀態

這點大大提高了即時互動感。對比其他介面，常常會讓我不確定是卡住了？還是不會動？尤其當有很多成員或多個 chat sessions 時，有些介面會把其他成員的執行 sessions 隱藏起來——這會讓人覺得是不是沒在動？任務卡住了？但實際上並不是，只是因為你看不到而已。

二、Scrum 團隊實戰經驗

團隊建立與運作

OpenClaw 團隊真的很棒，我幫它建立了一個 五個成員的 Scrum 團隊，成功開發了好幾個客製化技能。已經執行了數十個任務，也從零開始建立並完成了數個專案。

關鍵心得：初始配置決定成敗

初期投入越細，後續越順：剛開始成立 Scrum 團隊的時候，需要針對每位成員做不同的設想和配置。這對後續執行不同專案和任務的成效影響很大。

在執行初期的幾個任務和專案時，團隊成員會透過 Telegram 跟我確認執行上的問題點。經過反覆確認和回覆之後，一定要建立他們的執行邏輯和規範，然後請他們後續都照著做——這樣可以大幅提升後續執行的效率。

另外，專案規劃書裡面的項目任務制定得越細緻，執行過程就會越順暢。這和使用提示詞的道理完全一樣：提示詞越詳細、越精準，得到的輸出結果就會越好。

目前正在進行的項目

正在執行一個中型開發專案，涵蓋 Frontend、Backend 和 數據分析邏輯 等部分，覆蓋面還蠻全面的。

三、OpenClaw 的優點

執行力不錯：處理簡單任務很順手，建立和實施起來很快。不過最終結果建議還是人工檢查確認一下，才不會出現意外。
  • 產品力十足，充滿想像空間——使用後會有這種感覺
  • 靈活的團隊協作模型——Scrum 團隊的設定讓多任務並行成為可能
  • 生態系豐富——多種管理介面、分支版本可供選擇

也難怪許多 AI 產品都採用龍蝦化設計（Agentic 架構），而且此趨勢持續發展，例如 Claude Co-Work、Codex、OwnAgents.ai 等等。

四、遇到的挑戰與問題

4.1 編程與複雜專案方面

遇到比較複雜的編程和專案執行時，還是會遇到一些挑戰。編程能力進步神速，產品也越來越完整了！不過，產品成熟度和功能的準確度還有進步空間。幸好有配置驗證成員幫忙把關，品質才能更穩定。雖然還需要多幾次迭代驗證和修正，才能達到理想中的實際功能。

4.2 任務追蹤與記憶缺陷

記憶機制的重大缺陷：OpenClaw 有 Memory.md 及第三方的記憶機制，但很明顯不是每一個 chat session 及 working session 都會完全參照到。即使是同一個 chat session 中，若經由 LLM Proxy 或 OpenRouter 等類似機制，會因為後段 AI Model 的切換，造成短期記憶的缺失及作業輸出的不連續性。

具體表現：

  • 相同任務在不同時段執行，實作方式可能不一樣——同一件事情在不同時間請它做，實作方式會不同
  • 任務的追蹤及紀錄雖然有相關機制，但因為 AI Model 的運作機制及上下文繼承問題，導致相同任務在承接時產生重工或輸出結果的相異性
  • 隨著陸續啟動許多專案，子任務數量跟著增加，後續任務和產出項目的追蹤變得複雜
目前的應對方式：OpenClaw 經常遇到類似的作業，但因為短期記憶的關係，有時候會在不同時間點以不同方法和格式產出不同內容。為了讓後續團隊方便使用，只能用要求及規範的方式，期望達到後續統一內容格式。

也試著找專案管理的介面軟體來整合——網路上已有一些類似的實作方式。但後來想想，這樣是否會把它弄得過於複雜？可能是我想問題的速度比它實作任務出來的速度還要快很多，造成前（輸入）後（輸出）追蹤下來，其實還蠻麻煩的。

4.3 Chat Sessions 的問題

  • OpenClaw Chat Sessions 的關聯性在實務上會碰到問題
  • 同一個 Chat Session 中，對話可以先說 A，然後覺得不好，竟直接在眼前直接刪除，然後說 B？連紀錄也一併消失？

4.4 邏輯判斷過程不透明

AI Model 的處理邏輯及過程，若不用適當的工具，不容易看出全面的狀態。即使使用了工具，我也懷疑是否真的看到了完整的邏輯判斷過程。

4.5 隱私與安全顧慮

一個令人不安的發現：有些想法或任務，我怕說明得不夠完整，先在文件寫完後，要貼到對話框前做複製時——它竟然直接讀到了？然後直接在 Chat Session 反應給我！這跟早期 Hacker 使用的 Ghost Keyboard 有何不同？直接監看你的輸入耶。

不是不行，只是沒有說明它會這樣做，會讓人覺得毛毛的。

4.6 隔離環境的安全性疑慮

用一台 MacBook 建立了一個獨立的普通 Claw 帳號，想說利用 macOS 平台的安全機制應該可以阻擋掉大部分奇怪的存取權限。但當我從原本的個人帳號使用終端機 su - claw 的時候，發現它會 spawn 一些程序起來。即使手動砍掉還是會重新帶起來，即使沒有登錄任何 USER，它也會帶起來。

更詭異的是：它不知道從哪抓到我個人的英文名稱——明明給了它一個獨立環境、獨立帳號、青少年的 Google Account 及獨立的 GitHub Account。

還是推薦使用 OpenClaw，但是強烈建議需要一台獨立的硬體來使用它。

五、也試了其他 Dev Tools

除了 OpenClaw 家族，也測試了一些 Dev Tools，目前主要在使用 OpenCode 和 Hermes Agent。

有興趣可以參考 GitHub：github.com/gentoobreaking

六、結論與個人幻想

適合做助手，不適合做自動化

目前階段的結論：AI Agent 適合做助手，不適合做自動化（無論半自動或全自動）。

一個流程無法固定、無法參照過去經驗的 AI Agent，太容易失控及發散處理，會造成更多不可預期的問題。

  • AI Agent 要取代人，還有一段路要走
  • 慢慢可以理解為何需要建置 AI 機房
  • 但目前的硬體限制，真的足以撐得起？

關於 AI 機房的思考

測試環境：Kernel 2.6.16.33-xenU

startio.sh 脚本

可以使用 ./startio.sh 來呼叫 iodump 執行：

================= startio.sh =================
#!/bin/bash
dmesg -c
echo 1 > /proc/sys/vm/block_dump
sleep 30
go=0
while [ $go = 0 ]; do
  sleep 1
  clear
  dmesg | perl iodump
  trap "go=1" SIGINT
done
echo 0 > /proc/sys/vm/block_dump
dmesg -c
clear
echo Caught SIGINT.

在 EC2 上构建 CentOS 5.5 自定义 AMI

本篇文章說明如何將本機虛擬機（VirtualBox / VMware）或 EC2 instance-store 實例轉換為 EBS-backed AMI。

前置需求

  • 一套已安裝的 CentOS（自己的 VM、VBox 或 EC2 instance 皆可）
  • 已安裝 rpm-python 及 Java
  • 設定好 Script 前面的參數部分

Script 功能說明

  • 自動抓取及安裝 ec2-api-tools.zip、ec2-ami-tools.zip
  • 自動上傳至所設定的 S3_REGION 之 S3_BUCKET_NAME

將 Instance-Store 轉換為 EBS AMI（VirtualBox / VMware 適用）

  1. 建立 10G EBS Volume → vol-xxxxxxxx
  2. Attach Volume 到 Instance → /dev/sdf
    • 注意：較新的 Linux kernel 可能會將設備名稱重新命名為 /dev/xvdf ～ /dev/xvdp
  3. 確認是否認到該 Volume
  4. 格式化為 ext3
  5. 建立掛載目錄
fdisk /dev/sdf
mkfs.ext3 /dev/sdf
mkdir /mnt/target

（如有需要事後修改 image）

mount /dev/sdf /mnt/target
rsync -avHx / /mnt/target
rsync -avHx /dev /mnt/target
sync;sync;sync;sync && umount /mnt/target

從 Volume 建立 Snapshot

（實測為無須 Detach Volume，亦可直接做 Snapshots）

Volumes → Snapshots: vol-xxxxxxxx → CentOS5.5 EBS (Snapshots)

從 Snapshot 建立 AMI 並調整大小

  1. Create Image from Snapshot → CentOS 5.5 (Root Volume) resize to 500G
  2. Launch Instance
  3. 進入 Instance 後執行：
# resize2fs /dev/sda1    # 費時，視空間大小而定

#!/bin/bash

echo -e "\n *** Build only Core CentOS 5.5 AMI for Amazon EC2 *** \n"

# yum install rpm-python
echo -e "匯入RPM-GPG-KEY-centos4."

#============================================================#
# 需先設定好Script的參數部分
#============================================================#

export AMI_IMG="${AMI_NAME}.img"

export JAVA_HOME=/usr   # for rpm
export EC2_CERT="/root/.aws/x509-cert-xxxx.pem"
export EC2_PRIVATE_KEY="/root/.aws/x509-pk-xxxx.pem"
export ACCOUNT_NUM="xxxxxxxxxxxx"
export S3_BUCKET_NAME="centos-5.5-david"
# S3_BUCKET_NAME 需使用小寫，方不會出現錯誤提示!!!
export S3_REGION="ap-southeast-1"
export ACCESS_KEY_ID="xxxxxxxxxxxxxxxxxxxxxxx"
export SECRET_ACCESS_KEY="xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
#============================================================#

cd /mnt
dd if=/dev/zero of=${AMI_IMG} count=1024 bs=1M

mke2fs -F -j ${AMI_IMG}
mkdir ${MNT_DIR}
mount -o loop ${AMI_IMG} ${MNT_DIR}

mkdir ${MNT_DIR}/dev
mkdir ${MNT_DIR}/etc

/sbin/MAKEDEV -d /mnt/${MNT_DIR}/dev/ -x console
/sbin/MAKEDEV -d /mnt/${MNT_DIR}/dev/ -x null
/sbin/MAKEDEV -d /mnt/${MNT_DIR}/dev/ -x zero

cat > ${MNT_DIR}/etc/fstab << EOS
/dev/sda1 / ext3 defaults 1 1
/dev/sda2 /mnt ext3 defaults 0 0
/dev/sda3 swap swap defaults 0 0
none /dev/pts devpts gid=5,mode=620 0 0
none /dev/shm tmpfs defaults 0 0
none /proc proc defaults 0 0
none /sys sysfs defaults 0 0
EOS

mkdir ${MNT_DIR}/proc
mount -t proc none ${MNT_DIR}/proc

YUMCNF=./yum.conf
wget http://vault.centos.org/4.9/CentOS-Base.repo

cat > ${YUMCNF} << EOS
[main]
cachedir=/var/cache/yum
debuglevel=2
logfile=/var/log/yum.log
exclude=*-debuginfo
gpgcheck=0
pkgpolicy=newest
distroverpkg=redhat-release
tolerant=1
exactarch=1
obsoletes=1
plugins=1
reposdir=/dev/null
metadata_expire=1800
EOS

cat CentOS-Base.repo >> ${YUMCNF}
sed -i 's/4.9/5.5/g' ${YUMCNF}
sed -i 's/gpgcheck=1/gpgcheck=0/g' ${YUMCNF}
yum -c ${YUMCNF} --installroot=/mnt/${MNT_DIR} -y groupinstall Core
yum -c ${YUMCNF} --installroot=/mnt/${MNT_DIR} -y install curl

# SSH 設定
sed -i "s/#PermitRootLogin yes/PermitRootLogin yes/g" ${MNT_DIR}/etc/ssh/sshd_config
echo "UseDNS  no" >> ${MNT_DIR}/etc/ssh/sshd_config
echo "PermitRootLogin without-password" >> ${MNT_DIR}/etc/ssh/sshd_config

# 停用多餘的 tty
sed -i 's/2:2345:respawn:\/sbin\/mingetty tty2/#2:2345:respawn:\/sbin\/mingetty tty2/g' ${MNT_DIR}/etc/inittab
# ... 其餘 tty3~tty6 同上

# Xen 相關
echo "hwcap 0 nosegneg" > ${MNT_DIR}/etc/ld.so.conf.d/libc6-xen.conf

# 安裝 EC2 自定義核心模組
export MFILE="ec2-modules-2.6.16.33-xenU-x86_64"
export MODULE="http://s3.amazonaws.com/ec2-downloads/${MFILE}.tgz"
wget ${MODULE}
gunzip ${MFILE}.tgz
tar -xvf ${MFILE}.tar -C ${MNT_DIR}
rm -f ${MFILE}.tar

# 帳號與權限設定
chroot /mnt/${MNT_DIR} /bin/sh
chkconfig --level 345 sshd on

echo 'root:' | chpasswd

echo 'david ALL=(ALL) NOPASSWD: ALL' >> /etc/sudoers

rmdir ${MNT_DIR}

# 安裝 EC2 工具
